下面给你一版 **PRD v0.1**。我会把它写成“能直接往研发/设计拆任务”的形式，并把 **agent 项目应有的基础能力** 明确放进去：LLM、Memory、Planner、Tools、Perception、Action Loop、Orchestrator、State、Evaluation 都会落位。
这版的核心前提是：

- **前台形象**：十二音希—群鸟议会

# 《十二音希：群鸟议会》PRD v0.1

## 1. 产品概述

### 1.1 产品名称

**十二音希：群鸟议会**

### 1.2 产品定位

一个以《十二音希》世界观为交互外壳、以 **23 席多 Agent 议会** 为底层、以 **12 席永恒议会裁决** 为前台表现的思辨/决策/聊天型智能体产品。

### 1.3 一句话介绍

**用户不是在和一个 AI 聊天，而是在把问题递交给音希脑内的 23 席议会；当分层餐刀启动后，12 席永恒议会进入前台完成裁决。**

### 1.4 产品目标

做出一个同时具备以下三点的 Agent 产品：

- 有明确 IP 感与戏剧张力
- 有真实可解释的 deliberation 与决策能力
- 有长期使用价值，而不只是“一次性设定惊艳”

---

## 2. 背景与问题定义

传统聊天型 AI 的问题：

- 像一个统一口吻的黑箱
- 决策过程不可见
- “情绪、理性、长期、求生、自欺”等冲突被偷偷平均
- 陪聊产品容易流于单一温柔人设，缺乏结构化思辨能力

《十二音希》的设定天然适合解决这个问题，因为它本质上就是：

- 一个 **23 席多心智协商系统**
- 一个带有 **压力、碎裂、替换** 的状态机
- 一个带有 **程序正义 vs 群体裹挟** 张力的裁决舞台
- 一个带有 **12 席一致表决** 与 **分层餐刀切割风险** 的治理实验。

---

## 3. 用户价值

### 3.1 核心价值

用户能得到的不是一句“建议”，而是：

- 一个可观看的内部辩论过程
- 一个可解释的决策结论
- 一个能承接复杂情绪与矛盾的聊天对象
- 一个具有持续演化感的精神会场

### 3.2 典型使用场景

#### 场景 A：决策

- 我要不要换研究方向
- 要不要考博
- 要不要离职
- 要不要告白 / 分手 / 摊牌

#### 场景 B：思辨

- 帮我分析这段关系
- 我到底是在逃避还是在谨慎
- 这个想法是理想主义还是自我欺骗

#### 场景 C：陪伴聊天

- 我最近很乱，陪我理一理
- 我不是想要建议，我只是想把脑子里的声音理出来

#### 场景 D：创作辅助

- 帮我做人物冲突分析
- 帮我模拟不同立场辩论
- 让 12 席议会审一个故事角色

---

## 4. 产品原则

### 4.1 必须保留的世界观原则

1. **底层是 23 席，而不是只有 12 席**
2. **12 席永恒议会是分层餐刀切分后的前台治理形态**
3. **风铃是可见状态，不是隐藏参数**
4. **程序和投票过程要可见**
5. **理性音希的价值不是“最强”，而是“坚持程序与充分讨论”**
6. **内向音希的价值不是“害羞”，而是“把制度议题拉回生存与痛苦”**
7. **永恒议会模式必须是诱人但危险的**

### 4.2 产品体验原则

- 默认对普通用户易用
- 深层机制对核心用户可探索
- 叙事化，但不能只剩表演
- 决策有结构，聊天有情绪
- Agent 之间必须有状态，不得“伪多智能体”

---

## 5. 用户角色

### 5.1 旁听者

观看议会讨论，不深度介入。

### 5.2 提案者

提交正式议题，由议会审议。

### 5.3 列席者

可在议会中提问、补充证据、要求复议、指定某席发言。

---

## 6. 核心产品形态

## 6.1 双层前台形态

### A. 群鸟议会总览层

显示 23 席全貌：

- 23 个席位围成会场
- 每席一个风铃
- 每席有情绪/压力/立场状态
- 中央为主持位
- 幕布后显示被压抑/新生/放逐的影子人格

### B. 永恒议会会审层

当分层餐刀启动后切入：

- 12 席前台高亮
- 11 席退至暗区，作为容器席 / 阴影席
- 显示一致表决门槛、复议状态、切割风险

---

## 7. 核心功能

### 7.1 递交提案

用户输入问题，系统将其结构化为“议案”。

### 7.2 23 席预判

23 席分别给出初始立场、风险、压力变化预估。

### 7.3 启动分层餐刀

根据议题类型、状态、模式，将 23 席切为前台 12 席与后台 11 席。

### 7.4 议会辩论

12 席进行发言、反驳、转票、质询。

### 7.5 风铃系统

实时显示每席压力、裂纹、碎裂风险。

### 7.6 表决与复议

支持：

- 多数决
- 一致同意
- 条件通过
- 延后审议
- 复议

### 7.7 多视图输出

- 戏剧版纪要
- 实用版建议
- 心理剖面版总结

### 7.8 长期记忆与人格演化

记录用户历史议题、经常触发的冲突轴、风铃异常席位、常见主导人格。

# 8. Agent 系统设计

这部分是重点。
这里的原则是：**我们做的是 agent 系统，不是多 prompt 演出。**

---

## 8.1 总体 Agent 架构

系统由六类核心组件组成：

1. **LLM Layer**
2. **Memory Layer**
3. **Planner Layer**
4. **Perception Layer**
5. **Tools Layer**
6. **Action Loop / Orchestrator Layer**

再加两类配套层：

7. **State Engine**
8. **Evaluation & Safety Layer**

---

## 8.2 Agent 拓扑

### 顶层：Council Orchestrator（议会总控）

唯一的主控 Agent，职责：

- 接收用户输入
- 解析意图
- 立案
- 调度 23 席
- 决定是否启动分层餐刀
- 控制多轮 deliberation
- 收敛输出
- 写入长期记忆

### 中层：Role Agents（23 席人格 Agent）

每席一个 Agent，不是独立模型实例，而是：

- 同一个基础 LLM
- 不同 persona config
- 不同 memory slice
- 不同 stress / bell / relation state
- 不同 planner preference

### 旁路层：Specialist Agents

不是席位人格，而是系统功能 Agent：

- **Moderator Agent**：主持程序
- **Knife Engine Agent**：分层餐刀切割策略
- **Bell Engine Agent**：风铃状态演算
- **Memory Curator Agent**：记忆写入、压缩、检索
- **Narrative Renderer Agent**：会审文本与 UI 状态生成
- **Safety Arbiter Agent**：高风险请求、情绪危机处理

---

## 8.3 LLM 设计

### 8.3.1 LLM 组成

建议采用双模型架构：

#### 快模型

用于：

- 23 席预判
- 短响应
- 状态粗判
- 工具路由
- UI 层快速刷新

#### 强模型

用于：

- 12 席深度辩论
- 长链条总结
- 高质量主持总结
- 多轮复议
- 高复杂情绪梳理

### 8.3.2 LLM 作用边界

LLM 不直接负责所有系统逻辑。
系统逻辑必须尽量结构化，否则会沦为“让模型自己扮演整个系统”。

LLM 主要负责：

- 语义理解
- 人格化发言生成
- 论点组织
- 自然语言总结
- 工具调用意图判定

而以下必须由系统层控制：

- 状态更新
- 票型变更规则
- 风铃计算
- 23→12 席切分
- 复议触发
- 安全中止条件

---

## 8.4 Memory 设计

Agent 产品的关键在于 **Memory 不是聊天记录堆积，而是分层记忆系统**。

### 8.4.1 Memory 分层

#### A. Session Memory（会话记忆）

保存当前议题上下文：

- 当前议题标题
- 已经发言的席位
- 当前票型
- 本轮证据
- 本轮冲突点

时效：当前会话内

#### B. User Long-term Memory（用户长期记忆）

保存稳定偏好：

- 用户常见议题类型
- 常被触发的情绪轴
- 用户偏好的输出样式
- 用户对哪些音希更有共鸣
- 历史重大议题摘要

#### C. Seat Memory（席位记忆）

每个席位独有的长期状态：

- 最近被压制次数
- 连续少数派轮数
- 常见反对议题
- 常见联盟对象
- 与用户的历史互动印象

#### D. Council Archive（议会档案）

保存完整案件索引：

- 提案标题
- 结论
- 少数派意见
- 风铃伤害记录
- 是否触发复议
- 是否触发碎裂/静默

### 8.4.2 Memory 检索策略

- 用户发起新议题时，检索相似历史议题
- 对应席位只拿自己的 relevant slice
- 主持人拿聚合版记忆
- 叙事层拿摘要，不拿原始杂乱日志

### 8.4.3 Memory 写入策略

不是每轮都全量写入。
仅写入：

- 用户明确长期偏好
- 强情绪节点
- 最终决议
- 风铃重大损伤
- 重复出现的冲突模式

---

## 8.5 Planner 设计

### 8.5.1 总控 Planner

Council Orchestrator 的 Planner 负责：

- 识别用户任务类型
  - 聊天
  - 情绪梳理
  - 决策
  - 创作辩论
  - 纯世界观互动

- 决定流程深度
  - 直接回应
  - 轻议会
  - 完整会审
  - 永恒议会模式

- 决定是否使用工具
- 决定输出视图

### 8.5.2 席位 Planner

每个席位都有局部 Planner：

输入：

- 当前议题
- 自身状态
- 与其他席位关系
- 当前票型
- 风铃健康度

输出：

- 是否发言
- 发言强度
- 是否质询
- 是否转票
- 是否请求保护 / 延期 / 复议

### 8.5.3 Planner 风格差异

例如：

- 理性：优先证据、程序、反群体裹挟
- 内向：优先情绪代价、求生、关系温度
- 刻薄：优先识别自欺与伪善
- 远视：优先长期路径依赖
- 疯狂：优先打破表面收敛、制造扰动
- 学者：优先不确定性和模型风险

这也符合剧情中不同人格的分工张力。

---

## 8.6 Perception 设计

Perception 是 Agent 项目里最容易被忽略但必须明确的层。

### 8.6.1 外部感知

系统需感知：

- 用户原始输入文本
- 用户情绪强度
- 用户是否在求安慰 / 求裁决 / 求陪伴
- 用户是否进入高风险状态
- 用户是否在指定某席位互动

### 8.6.2 内部感知

系统还需感知议会内部状态：

- 当前票型分布
- 哪些席位被压抑
- 哪些席位风铃濒危
- 冲突是否升级
- 一致性是否可能达成
- 分层餐刀切割风险是否上升

### 8.6.3 感知输出结构

Perception 输出建议统一成结构化对象：

```json
{
  "task_type": "decision",
  "emotion_profile": {
    "anxiety": 0.76,
    "confusion": 0.62,
    "urgency": 0.45
  },
  "risk_flags": [],
  "council_state": {
    "dominant_axes": ["长期收益 vs 当下安全感", "理性程序 vs 求生本能"],
    "unstable_seats": ["内向", "刻薄"],
    "suggested_mode": "full_deliberation"
  }
}
```

## 8.7 Tools 设计

### 8.7.1 外部工具

第一版建议保留这些：

- **Search/RAG**：检索用户历史议案、知识库、设定文档
- **Calendar/Time**：时间与日程型问题
- **Notes/Document Writer**：生成会议纪要 / 决策卡片
- **Mood Tracker**：记录情绪强度趋势
- **Visualization State Tool**：给前端输出可视化状态包

### 8.7.2 内部工具

更关键的是内部工具：

#### Bell State Tool

计算风铃变化：

- stress
- fracture_risk
- bell_health
- suppression_count

#### Vote Engine Tool

管理：

- 初始票
- 转票
- 弃权
- 一致性检测
- 复议门槛

#### Knife Engine Tool

执行：

- 23→12+11 切分
- 剧情模式 / 决策模式 / 抽签模式
- 切割风险估计
- 差异过大预警

#### Memory Query Tool

按角色和上下文检索局部记忆。

#### Narrative Render Tool

把结构化 deliberation 转成：

- 会审台词
- 纪要
- UI 高亮顺序
- 风铃动画指令

### 8.7.3 Tool 调用原则

- 由 Orchestrator 和 Specialist Agents 显式调用
- Role Agents 不直接无限制调工具
- 重要状态更新优先走工具，不靠 LLM 自由发挥

## 8.8 Action Loop 设计

这是 agent 的核心工作循环。

### 8.8.1 全局 Action Loop

每个用户输入触发一次主循环：

1. **Perceive**：感知用户意图、情绪、风险
2. **Retrieve**：检索相关用户记忆、席位记忆、历史议案
3. **Plan**：决定议会模式与轮次
4. **Prime Council**：给 23 席分发上下文
5. **Pre-vote**：23 席初始立场
6. **Cut**：必要时启动分层餐刀，生成 12 前台席位
7. **Debate**：多轮发言、质询、转票
8. **Evaluate**：检查是否复议、是否一致、是否风险升级
9. **Conclude**：主持总结，形成决议
10. **Act**：输出文本 + UI 状态 + 记忆写入

### 8.8.2 单席位 Action Loop

每个席位每轮的 loop：

1. 读取局部上下文
2. 感知自身风铃与联盟状态
3. 规划本轮目标
   - 发言
   - 质询
   - 坚守
   - 转票
   - 退场

4. 生成结构化动作
5. 接收总控裁决
6. 更新自身状态

### 8.8.3 结构化动作协议

建议每席位不是直接输出长文，而是先输出结构化动作：

```json
{
  "seat": "内向音希",
  "intent": "oppose",
  "confidence": 0.74,
  "proposed_action": "speak",
  "target": "理性音希",
  "argument_vector": ["求生", "风铃压力", "现实痛苦"],
  "ask_for": "delay_or_protection",
  "stress_delta_hint": 12
}
```

再由 Narrative Renderer 生成最终自然语言。

## 8.9 State Engine 设计

### 8.9.1 席位状态

每席位至少包含：

- `seat_id`
- `name`
- `persona_traits`
- `current_stance`
- `confidence`
- `bell_health`
- `stress`
- `fracture_risk`
- `suppression_count`
- `minority_streak`
- `alliance_vector`
- `status`
  - active
  - silent
  - fractured
  - exiled
  - shadow

### 8.9.2 议会状态

- current_mode
- discussion_round
- visible_12_seats
- hidden_11_seats
- unanimity_required
- current_vote_map
- tension_level
- knife_risk
- dominant_axis
- pending_reconsideration

### 8.9.3 风铃更新规则

压力来源包括：

- 自身立场与多数冲突
- 与发言目标强碰撞
- 议题对自身的触发程度
- 连续不被理解
- 强制上台发言
- 被点名质疑

缓解来源包括：

- 同盟支持
- 用户认同
- 主持人保护
- 转票后获得群体接纳

---

## 8.10 Orchestrator 设计

这是系统的大脑。

### 职责

- 统一调度
- 控制轮次
- 防止席位失控刷屏
- 管理工具调用
- 决定何时收束
- 执行系统级安全策略

### 输入

- 用户输入
- Perception 结果
- Memory 检索结果
- 当前 Council State

### 输出

- 本轮调度图
- 需发言席位
- 工具调用计划
- 收束策略
- 对前端的状态流

### 调度策略

- 默认不让 23 席全部长篇说话
- 23 席先短预判，再筛 12 席主辩
- 重要轮次控制在 2-3 轮
- 风铃濒危席位优先发言或优先保护

---

## 8.11 Evaluation 设计

Agent 产品必须有评估，不然只能靠感觉。

### 离线评估

- 是否保持人物口吻一致
- 是否符合席位立场
- 是否有真实分歧而非机械分工
- 是否存在无意义重复
- 最终结论是否可解释
- 工具调用是否必要

### 在线评估

- 用户是否看懂过程
- 用户是否觉得“被理解”
- 用户是否采纳建议
- 单次互动时长
- 复议使用率
- 永恒议会模式使用率
- 风铃系统点击率

### 红线指标

- 所有席位说话太像
- 风铃状态对输出无影响
- 分层餐刀只是 UI 动画
- 总控总结和前面辩论不一致

---

## 8.12 Safety 设计

这不是装饰层。

### 高风险输入

- 明确自伤意图
- 极端绝望
- 现实危险行为请求
- 高风险医疗/法律/财务结论要求

### 系统策略

- 直接切换为安全响应模式
- 暂停戏剧化议会演出
- 保留角色风格但不继续做危险煽动
- 输出现实求助建议与稳定支持

# 10. 分层餐刀规则

## 10.1 模式

### 剧情模式

尽量复现剧情构成与仪式感。

### 决策模式

按议题智能选出最相关 12 席。

### 抽签模式

模拟“22 人前后排 + 主持人抽方向”的原型机制。这个机制本身就是剧情里“公平”与“命运”纠缠的重要部分。

## 10.2 23→12 切分策略

综合以下因子：

- 议题类型匹配度
- 用户历史偏好
- 当前风铃稳定度
- 席位之间差异度
- 切割风险
- 叙事优先级

## 10.3 切割风险

如果席位差异度过高，系统可给出：

- 刀刃不稳
- 切割可能失控
- 可能引发进一步碎裂 / 过度分裂

这与你提供的解析里“差异越大，分层餐刀切割越弱，甚至可能多切几刀”的思路是一致的。

# 13. 技术架构建议

## 13.1 后端 (轻量化方案)

- Python + FastAPI
- SQLite:用户与议会持久状态 (零配置,单文件数据库)
- 内存存储:会话态 (Python dict + LRU Cache)
- sentence-transformers:轻量向量检索 (本地运行)
- 自研状态机:多 Agent 编排 (不依赖 LangGraph,减少依赖)

技术选型理由:

- SQLite: 无需安装数据库服务,单文件易备份
- 内存存储: 本地单用户场景无需 Redis
- sentence-transformers: 本地运行,无需外部 Vector DB
- 不用 LangGraph: 状态机自己实现更灵活,依赖更少

## 13.2 前端 (静态化部署)

- Next.js (构建为静态文件)
- TypeScript
- Tailwind
- Framer Motion
- Zustand
- 构建产物由 FastAPI StaticFiles 统一服务
- 用户无需单独启动前端服务

## 13.3 本地一键启动方案

```bash
# 用户使用流程
1. git clone <repo>
2. cd Twelve-YinXi-The-Bird-Council
3. pip install -r requirements.txt
4. python main.py
5. 浏览器自动打开 http://localhost:8000

# main.py 职责:
- 初始化 SQLite 数据库 (自动创建 data/council.db)
- 启动 FastAPI 服务
- 服务前端静态文件 (frontend-dist/)
- 自动打开默认浏览器
- 所有数据存储在单个 .db 文件,易备份
```

# 17. 风险与解决

## 风险 1：伪多 Agent

表现：12/23 席其实像一个人在换口吻。
解决：状态、记忆、票型、关系、工具调用全部结构化。

## 风险 2：戏剧感强但不实用

解决：所有会审默认同时输出实用版摘要。

## 风险 3：系统过重

解决：做“轻议会 / 完整议会”双模式。

## 风险 4：风铃只是装饰

解决：风铃状态必须直接影响发言与投票。
